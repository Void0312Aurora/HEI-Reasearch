你这组 probe 结果的解读大体正确：PMI 已经“开始改变几何”，但目前局部邻域仍主要由词林骨架（树结构）决定；因此“情绪词看起来更好、具体名词更差”是一个合理的症状组合。不过，你给出的三条改进建议里，有两条我认为是“必须先做的诊断/清洗”，否则直接加权重或加步数很可能只是在放大噪声。

一、先把“观测口径”修正，再谈模型是否真的弱

1. 过滤 Code 节点不仅是展示层问题，而是邻域定义问题
   你看到大量 `<Code:...>`，说明“类别/中间节点”作为粒子参与了近邻。对于对话与语义 probe，这会系统性污染邻域：中间节点在树结构上天然“近”，会把词节点的局部几何拉向“结构中心”，导致你看到的“猫→昂贵/美美/冷静”这种跨域漂移。
   建议你做两件事（优先级很高）：

* Probe 输出：只显示 Word 叶子节点（不足 K 个就用“展开 Code→叶子词”补齐）。
* 训练/检索层：把 Code 节点从“可被 KNN 检索的候选集合”中剔除（它们可以存在于系统里用于约束，但不参与 readout / nearest-neighbor probe）。

如果你只做展示过滤、不改候选集，你仍会被 Code 节点在几何中“挤占邻域容量”影响结果。

2. 对“猫/医生/红”这类名词，先确认 PMI 覆盖是否真的存在
   在你们当前的构图流程（Top-K、c_min、停用词过滤）下，名词的 PMI 邻居可能因为分词、别名或频次阈值被削弱。例如：

* “猫”在语料里常以“猫咪/小猫/猫猫/喵”等出现，若未统一归并，你的 PMI 边会很稀。
* “红”是高歧义词（颜色、姓氏、红利/走红/红色政权等），PMI 会非常分散，且容易被树结构稀释。

建议做一个极快的核验：对每个 probe 词，直接查 PMI 图的邻接表（Top-50 PMI neighbors 是否合理、边权是否显著），不要通过几何距离间接判断。
如果 PMI 邻接本身就弱或为空，加权重/加步数不会产生你想要的联想。

二、关于“加 PMI 权重 0.5→0.8”与“加训练步数”的评价

* 权重继续上调不是不可以，但应当是“最后一步”，而不是第一步。原因是你们已经看到 hub 风险并刚治理过；权重继续上升最容易把“新一轮隐性 hub（过泛内容词、结构词）”重新顶起来，Residual 也可能再抬头。
* 步数延长更安全，但只有在 PMI 覆盖足够且候选集合去除了 Code 节点后，延长步数才会把 PMI 的效应“沉淀”到整体几何里。

更稳的策略是：保持 0.5，先把 PMI 图质量与 readout 口径修正；如果 probe 显著改善但仍不达标，再在 0.5→0.65→0.8 做小步扫参，并用 Gate C/N/H 做硬闸门。

三、我同意“Definition Edges”是解决名词类 probe 的关键，但要用对方式

对“猫→动物”“医生→医院/治疗/病人”这种联想，PMI 能做，但效率不高且受语料分布影响大。定义句/释义边对抽象关系非常有效，尤其当你们的骨架是词林（更偏同义场）时，定义边能补齐“属性/上位/用途”的关系类型。

建议的最小定义边做法（高性价比）：

* 从中文百科首句/词典释义提取：X 是一种/指/属于 Y…
* 只保留能映射到词林叶子的关键词 Y（或先映射到类别再展开到叶子）
* 权重设为介于 Skeleton 与 PMI 之间（通常比 PMI 略高），并对高歧义词（如“红”）做更强的消歧（只保留最常见释义路径）。

四、对你当前结论的修正表述

“PMI 边力度不足以主导局部邻域”这句话可能成立，但在做完两项必要修正之前（剔除 Code 候选、核验 PMI 覆盖），更准确的表述是：
